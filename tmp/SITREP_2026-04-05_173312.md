## What's BROKEN / Not Working

### CRITICAL: Video is a photo, not generated video

**THE #1 PROBLEM.** The render loop uses `FaceWarper.warp_mouth()` which shifts pixels on a static photo of Santa. This produces a barely-perceptible twitch, not video. The user correctly identified this as fundamentally wrong.

The **trained Gaussian Avatar** renders at 731 FPS and produces actual 3D-rendered faces. It MUST replace the FaceWarper in the render loop. The code change is straightforward:

```python
# CURRENT (broken — pixel shifting on photo):
frame = _face_warper.warp_mouth(mouth_open, head_tilt)

# NEEDED (3D model rendering):
renderer = GaussianAvatarRenderer.load_trained("santa/gaussian_avatar.pt")
frame = renderer.render_frame(expression=expression_params)["image"]
```

But the Gaussian model was trained with only 5000 iterations and produces blurry output (see `/tmp/santa_gaussian.jpg`). It needs either:
- More training iterations (50K+)
- Higher tex_size (256 instead of 128)
- LPIPS perceptual loss added
- Better camera calibration during training

### CRITICAL: Audio routing / Whisper hallucination

The bot hears "Thank you" repeatedly — Whisper hallucinates on quiet PulseAudio TCP audio. The bot's Chrome runs on Xvfb but its audio still goes through `output` sink which the bot also monitors. Need dedicated PulseAudio sink for the bot's Chrome.

### HIGH: Clone model only uses 4 of 9 embeddings

Visual (1152), Semantic (768), Emotion (3), Prosody (12) are extracted and trained. Missing:
- Speaker (WavLM 512-dim) — needs audio clip embedding
- Voice (ECAPA-TDNN 2048-dim) — needs voice profile extraction
- OCR Context — from MCP service
- Memory — from RuFlo
- Sentence (MiniLM 384-dim) — needs per-sentence embedding

### MEDIUM: Response latency

Gemma 4 E4B: 509ms (good). TTS: ~700ms warmed (good). But total pipeline from hearing to speaking is still ~3-5s due to 2s transcription window + audio normalization overhead.

## Architecture Decisions Made

### Video Delivery: Insertable Streams (CONFIRMED WORKING)
After trying: canvas captureStream (failed), Y4M FIFO (failed), v4l2loopback (Chrome can't see it), raw MediaStreamTrackGenerator (Meet rejects — no metadata) — the working approach is:
1. `--use-fake-device-for-media-stream` creates a track with real camera metadata
2. `MediaStreamTrackProcessor` reads the fake device frames
3. We replace the pixel content with our rendered frames
4. `MediaStreamTrackGenerator` outputs with spoofed metadata
5. Meet sees proper getSettings()/getCapabilities() → ACCEPTS the track

**This is saved in memory at:** `feedback_v4l2loopback.md` and `feedback_audio_delivery.md`

### Meeting Bot: Xvfb + Playwright (production approach)
Same architecture as tl;dv, Recall.ai, Otter.ai. Chrome runs headful on virtual display `:99`. No visible window. Separate browser session with own Google login.

### Chrome Extension: Thin Trigger
Extension detects Meet URL → POST to localhost:9877 → bot server launches Playwright. Extension does NOT record or inject — it's just a trigger.

### Clone Model: Perceiver + North Star Constellation
Multi-embedding Perceiver cross-attention with frozen identity anchors. Meaning-aware training with semantic consistency + cross-modal coherence loss. NOT just MSE on blendshapes.

### LLM: Gemma 4 E4B
Replaced Qwen3:8b. Apache 2.0, 128K context, native multimodal, 509ms inference.
